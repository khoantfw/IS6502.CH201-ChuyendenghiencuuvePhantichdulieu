# Are Counterfactual HR Intervention Scenarios Feasible?

## A Pilot Audit of Employees with High Predicted Attrition Risk

This research pipeline evaluates whether HR intervention scenarios can reduce the model's **attrition score** below the high-risk threshold using the IBM HR Analytics Employee Attrition & Performance dataset.

## Installation and Execution

The submitted artifacts were verified with **Python 3.14.3** and the pinned dependency versions in `requirements-lock.txt`. To reproduce the verified environment:

```bash
python3 -m venv .venv
source .venv/bin/activate
python3 -m pip install -r requirements-lock.txt
python3 run_pipeline.py
```

`requirements.txt` uses broader version ranges for development and includes Jupyter for running notebooks. Stochastic results from DiCE and XGBoost may vary across versions or platforms; use the lock file when reproducing the submitted artifacts.

## Data Preparation

The raw CSV is not distributed with the repository. Download the [IBM HR Analytics Employee Attrition & Performance dataset from Kaggle](https://www.kaggle.com/datasets/pavansubhasht/ibm-hr-analytics-attrition-dataset), then place the file at:

```text
data/raw/WA_Fn-UseC_-HR-Employee-Attrition.csv
```

Verify that the dataset matches the version used to produce the paper:

```bash
shasum -a 256 data/raw/WA_Fn-UseC_-HR-Employee-Attrition.csv
```

Expected SHA-256 checksum:

```text
a5c31e38bd7fafc9bc333884eb181b06b41b8e5e488e8f7ccb27199fb3be7659
```

On macOS, XGBoost may require OpenMP:

```bash
brew install libomp
```

If `libomp.dylib` is bundled with scikit-learn but Homebrew is unavailable, run:

```bash
DYLD_LIBRARY_PATH="$(python3 -c 'import pathlib, sklearn; print(pathlib.Path(sklearn.__file__).parent / ".dylibs")')" python3 run_pipeline.py
```

Research assumptions that can be varied for sensitivity analysis are defined in `config.yaml`; the fixed hyperparameters for the three candidate models are documented in `src/hr_recourse/data_modeling.py` and in the paper. For faster development runs, SHAP, DiCE, stability analysis, sensitivity analysis, or figure generation can be disabled individually in the `run` section. The default configuration enables all analyses reported in the paper.

When DiCE is disabled, Levels 1 and 2 are recorded as `skipped`. An employee without a feasible Level 3 scenario receives `analysis_status=partial` rather than being incorrectly classified as having no model recourse. Artifacts from a disabled phase are removed to prevent them from being mixed with results from an earlier run.

Run the tests:

```bash
python3 -m pytest -q
```

Build the paper after installing MacTeX or BasicTeX:

```bash

```

The paper uses the figures in `outputs/figures` directly, so run the pipeline before building the LaTeX document.

## Pipeline

1. Validate the data schema and quality, then create a stratified 60/20/20 split.
2. Fit logistic regression, random forest, and XGBoost using preprocessing learned only from the training set.
3. Select the model by validation average precision and choose the threshold that maximizes validation F2.
4. Report test metrics and identify test employees with `score >= threshold`.
5. Compute SHAP values for the original features and five HR feature groups.
6. Generate Level 1 and Level 2 DiCE scenarios with a threshold adapter, then rescore them using the original attrition score.
7. Evaluate an exhaustive Level 3 grid over `MonthlyIncome`, `StockOptionLevel`, and `OverTime`.
8. Audit feasibility, rank scenarios, and classify each employee.
9. Run seed-based stability checks and one-factor-at-a-time sensitivity analyses.

Primary output:

```text
outputs/tables/hr_intervention_feasibility_assessment.csv
```

Models, scenarios, supplementary tables, and figures are stored in `outputs/models`, `outputs/scenarios`, `outputs/tables`, and `outputs/figures`, respectively. Each scenario stores `changes_json` with the before-and-after values of every modified variable, allowing the profile to be reconstructed and scored independently.

## Scope and Responsible Interpretation

- The IBM dataset is **synthetic**. It is suitable for demonstrating the method but does not represent a real organization.
- `predict_proba` outputs are described as **attrition scores**; the pipeline does not claim that they are calibrated probabilities.
- Counterfactual explanations and SHAP describe model logic; they do not establish causal relationships.
- The 5%–30% salary-increase grid is a research search space, not an organizational policy or a real salary recommendation.
- The outputs must not be used to automate hiring, disciplinary, termination, promotion, or compensation decisions. Any real-world application requires human review, fairness assessment, privacy safeguards, legal compliance, and context-appropriate data.
